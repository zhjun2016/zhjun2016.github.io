列表数据错位

由于数据复用引起的，数据来源：1. 同步数据，2. 异步数据，

1.同步数据源

解决办法：充分考虑条件，对itemView中每个view设置默认值；

（itemBean为数据模型的字段，尽量使用字段记录view的状态）

2.异步数据源

场景：譬如ItemView 里面有 ImageView,通过一个url加载显示网络图片，等待服务器返回结果。假设一屏幕显示5条数据，滑动列表时，当前条目为第一条不可见时，第6条复用了第1条数据，导致图片显示错误。

解决：对异步的view设置tag,绑定url，在使用地方取出tag，判断一致才显示；

holder.ivHead.setBackground(R.drawable.place_holder);
//设置tag绑定异步地址
holder.ivHead.setTag(itemBean.imageURL);
//取出tag，判断
String url=(String)(holder.ivHead.getTag());
 if (TextUtils.equals(url, itemBean.imageURL)) {
    Glide.with(holder.ivHead).into(holder.ivHead).load(url);
  }

3.recyerlyView优化

3.1 设置ViewHolder不可复用（尽量不要此方法）

​ viewHolder.setIsRecyclable(false)

3.2 onBindViewHolder中优化，处理数据显示问题

  • 通过onBindViewHolder中的position或者viewHolder的getAdapterPosition来判断展示。

  • 如果是onBindViewHolder没有被调用，

    原因1：多布局类型，没重写getItemViewType导致`

    原因2：滑动缓存问题，

    当 RecyclerView 的条目很少的时候，比如只有六个，将 RecyclerView 从上滑动到下，这个时候是正常的，onBindViewHolder 会调用，不过此时从底部上划的时候，上方的 item 从不可见到可见的这个过程中，onBindViewHolder 并没有调用，这个时候我也就没办法进行一些刷新 item 的操作了。

    这个问题的原因是 onBindViewHolder 方法不调用导致的。

    此api去调整 RecyclerView 的复用逻辑和方式来解决 onBindViewHolder 没有调用，设置 setViewCacheSize 来调整 mCachedViews 保存 ViewHolder 的大小，就能解决问题。

​ 原因3：特殊情况不调用onBindViewHolder，可以监听 RecyclerView 的滑动，当滑动停止的时候，再调用 notify 刷新下列表也是可以的。此方法，经常优化用于滚动列表时，不更新数据。

3.3 使用异步回调时，改变 onBindViewHolder中view的状态，刷新时机不对；

在异步回来之前Adapter调用了notifyDataSetChanged()等刷新方法导致ViewHolder被刷新复用导致。如果是这种情况，需要保证避免异步回来之前刷新即可解决。
